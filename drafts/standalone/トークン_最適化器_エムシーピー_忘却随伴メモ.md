# token-optimizer-mcp 忘却随伴メモ

> 対象: `https://github.com/ooples/token-optimizer-mcp`
> 目的: token 圧縮器を HGK の忘却論にどう随伴させるかを定義する。

---

## 0. 核心

`token-optimizer-mcp` 全体を忘却随伴の核とみなすのは誤りである。

核に入るのは次の 2 操作だけである。

- `optimize_text`: 大きな文脈を cache key 付きの圧縮表現へ送る
- `get_cached`: cache key から原文脈を取り戻す

この 2 操作は、左手で荷物を畳み、右手で必要時に開く構造を持つ。
したがって、忘却論に随伴させるならこの対だけを芯として採る。

`smart_read` は便利だが芯ではない。
理由は、diff、truncation、chunk 返却を含み、元の文面と同型とは限らないからである。
それは「忘却射」ではなく、「忘却前の近似観測レイヤ」である。

---

## 1. 左右の役割

### 左随伴の役割

生の文脈を、そのまま保持せずに運用可能な圧縮表現へ送る。

直感:
重い書類束を封筒に畳み、封筒番号を付けて棚に戻す操作。

この左射が受け取る対象:

- 巨大ログ
- 一時的な build 出力
- 反復参照される API 応答
- 大きな補助資料

この左射が返すべきもの:

- cache key
- 原文 hash
- 生成時刻
- 対象の種類
- 復元条件

### 右随伴の役割

圧縮表現だけでは判断不能になったとき、原文を再展開する。

直感:
封筒番号だけでは証拠にならない場面で、棚から原書類を取り出して机に戻す操作。

右射が発動すべき場面:

- N-01「実体を読め」が優先される場面
- N-09「原典に当たれ」が優先される場面
- diff や要約では判断不能な場面
- 引用、契約、仕様、証明、監査などの高精度面

---

## 2. この随伴で保存したいもの

忘却は削除ではない。
忘却の目的は、認知面の占有を下げつつ、必要時の再接地能力を失わないことである。

この随伴で保存したいものは 4 つある。

| 保存対象 | 理由 | 最低限必要な保持物 |
|:---|:---|:---|
| 原文同一性 | 本当に同じものを戻せる必要がある | hash |
| 由来 | 何を畳んだか分からない圧縮は危険 | source path / source kind |
| 時間 | stale な復元は現実を壊す | created_at / freshness rule |
| 復元条件 | いつ raw に戻すかを明示する必要がある | restore contract |

---

## 3. 単位と余単位の直感

### 単位

生文脈を一度畳み、必要なときに戻した結果、元の判断面へ再接地できること。

図形的には、
大きな面を一度折り畳んでも、展開したとき折り線の位置が分かり、元の輪郭に戻れることに相当する。

この単位が失敗するのは次のときである。

- hash が欠落している
- 生成時刻が欠落している
- 元文脈が可逆圧縮ではなく近似出力に置換されている
- 復元要求が規則化されていない

### 余単位

一度復元した文脈を再度畳み直しても、余計な認知負荷や別物化を起こさないこと。

図形的には、
一度開いた地図をもう一度同じ折り目で畳み直せることに相当する。

この余単位が失敗するのは次のときである。

- 再圧縮のたびに key が乱立する
- 復元のたびに別 hash として扱われる
- 復元後に再び要約だけが残り、原文への道が消える

---

## 4. 忘却論としての採用境界

忘却論に採用してよいのは、次の条件を満たすものだけである。

1. 圧縮前後で原文 hash が追跡できる
2. 復元が機械的に行える
3. 復元要求の条件が言語化できる
4. SOURCE/TAINT を壊さない

この条件を満たさないものは、忘却随伴の芯ではなく、補助ヒューリスティクスに降格する。

そのため `smart_read` は次の位置づけになる。

| 機能 | 位置づけ |
|:---|:---|
| diff-only 返却 | 補助 |
| chunk-first 返却 | 補助 |
| truncation | 補助 |
| raw file 再取得 | 芯に接続可能 |

---

## 5. HGK での再構成

`token-optimizer-mcp` を忘却論へ随伴させるときは、外部ツール名をそのまま残す必要はない。
必要なのは機能の保存である。

再構成すべき機能は次の 3 層である。

| 層 | 役割 | HGK 側の候補 |
|:---|:---|:---|
| Forget | 圧縮して退避する | Mneme / project-index / artifact cache |
| Recall | 条件付きで戻す | Anamnesis / raw artifact restore |
| Gate | いつ raw に戻すか裁く | Horos N-01, N-09, N-10 |

つまり、これは単体で完結する道具ではなく、
Mneme と Horos のあいだに挟まる「可逆忘却層」として扱うのが正しい。

---

## 6. 採用しないもの

次は Oblivion の随伴核には採らない。

- installer による AI tool 全面設定改変
- hook 主導権の奪取
- `smart_read` の透明置換
- 「token が減るなら良い」という単純目的化

理由:
忘却論の主題は「何を手放しても自己同一性を保てるか」であって、
「どれだけ小さくできるか」ではないからである。

---

## 7. 研究課題

この随伴を Oblivion 側で育てるなら、課題は 3 つに分かれる。

| 課題 | 内容 |
|:---|:---|
| 可逆性 | どの操作を芯に入れ、どの操作を補助に落とすか |
| 恒常性 | 圧縮後も推論品質が壊れない条件をどう測るか |
| 復元契機 | どの Nomos が発火したら raw restore を強制するか |

---

## 8. 当面の結論

`token-optimizer-mcp` は、忘却論に対しては「圧縮器」ではなく「可逆忘却の局所実装」として読むべきである。

採るべき核:

- `optimize_text`
- `get_cached`
- hash / freshness / restore contract

採るべきでない核:

- `smart_read` の透明置換
- installer/hook takeover

したがって、Oblivion 側の随伴名は次でよい。

`Forgetting Cache Adjoint`

その中身は、
「何を忘れてよいか」を理論で定め、
「どう戻すか」を cache key で保証する二層構造である。
