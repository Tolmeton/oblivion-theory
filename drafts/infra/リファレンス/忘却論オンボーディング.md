## 4. まず押さえるべき3つの規則

### 4.1 番号の正本

系列番号の正本は `plans/出版計画_忘却論シリーズ.md`。  
この文書より先に番号を決めない。

### 4.2 記号の正本

記号・定理 ID・依存関係の正本は `drafts/infra/リファレンス/統一記号表.md`。  
新規稿で α や ρ を使うときは必ずここに合わせる。主張水準ラベル（定理/命題/仮説/予想/実験的知見）は §0.3 で定義されている。

### 4.3 昇格の規則

インキュベータ文書を番外編へ上げる条件は以下の3点。

1. 単独要旨が書ける  
2. シリーズ本文を前提にしない最小定義がある  
3. 参考文献と主張水準ラベルが自前で閉じる

満たさない文書は、即 `drafts/incubator/` に置くのではなく、`drafts/infra/`, `drafts/infra/リファレンス/`, `drafts/standalone/` のどれに属すかを先に判定する。`drafts/incubator/` は 3 本柱以外の常置場ではない。

## 4b. 執筆開始前チェック

新規論文の執筆、または既存論文の大規模改稿を行う前に、以下の4項目を確認する。

### ① 番号の正本確認
- `plans/出版計画_忘却論シリーズ.md` で系列番号が確定しているか確認する
- 番号が未決定の場合、執筆開始前に Tolmetes と番号を確定する
- **確認先**: 出版計画 §2 の正史表

### ② 記号の正本確認
- `drafts/infra/リファレンス/統一記号表.md` で使用する記号の定義を確認する
- 特に α, ρ, F, r, λ は複数の意味で使われている（§1.12 衝突記号表参照）。添字による曖昧性排除が必要
- 新しい記号を導入する場合は、記号表への追加を同時に行う
- **確認先**: 統一記号表 §1（主要記号）、§1.12（衝突記号解消）

### ③ 主張水準確認
- 各主張が定理/命題/仮説/予想/実験的知見のどの水準であるかを `drafts/infra/リファレンス/統一記号表.md` §0.3 に基づいて確認する
- 確信度ラベル (`[確信 90%]` 等) は `drafts/infra/リファレンス/批判反証レジストリ.md` §1b.2 のキャリブレーション規則に従う
- **確認先**: 統一記号表 §0.3（主張水準ラベル）、批判反証レジストリ §1b.2（確信度キャリブレーション）

### ④ 既存批判確認
- `drafts/infra/リファレンス/批判反証レジストリ.md` で該当論文に対する既知の批判と応答方針を確認する
- 横断批判（Z-系）が自分の論文に関係するかを確認する
- **確認先**: 批判反証レジストリ §1（横断批判）、§2（論文別批判）

## 6. 読者別ルート

### 🟢 新規執筆者ルート

> 初めて忘却論シリーズの論文を書く、または既存論文を大幅改稿する場合。

1. **本文書 §0–§4b** を通読する（所要: 10分）
2. `plans/出版計画_忘却論シリーズ.md` で番号と依存関係を確認する
3. `drafts/infra/リファレンス/統一記号表.md` §0–§1 で記号と主張水準ラベルを確認する
4. **§4b の4項目チェックを実行** してから執筆を開始する
5. 執筆中に新記号・新定理を導入した場合は、統一記号表への追加をその日のうちに行う

### 🟡 批判対応者ルート

> 外部査読・GPT 監査・ディベートで得られた批判に対応する場合。

1. `drafts/infra/リファレンス/批判反証レジストリ.md` §0 で運用規約を確認する
2. §1（横断批判）で自分の対応する批判が横断的かどうかを確認する
3. §2（論文別批判）で該当論文のエントリを確認する
4. §1b のシリーズ共通ルール（FEP claim strength / 確信度キャリブレーション / 圏論用語境界）を確認する
5. 応答を論文本文に反映した後、レジストリのエントリを更新する

### 🔵 統合作業者ルート

> モノグラフ統合、章間接続文の執筆、版管理など横断的作業を行う場合。

1. 本文書 §0–§5 を通読する（所要: 10分）
2. `drafts/infra/リファレンス/モノグラフ構成設計.md` で章カードと依存関係を確認する
3. `drafts/infra/リファレンス/統一記号表.md` §3（依存グラフ）と §3b（依存の型）で論文間の接続を確認する
4. モノグラフ構成設計 §3b の接続文設計原則（前章の獲得/次章への不足/橋渡し概念）に従う
5. 版管理の更新は、変更した文書のヘッダと、対応するインフラ文書の両方に反映する

### 参考: 旧ルート（数理中核・認知AI・全体像）

これらは「何を読むか」ではなく「どの順で論文を読むか」の道案内である。

- **数理中核から入る**: `論文I` → `論文II` → `論文III` → `論文VIII`
- **認知・AIから入る**: `論文IV` → `論文X` → `論文XI` → `論文XII`
- **全体像だけ先に掴む**: `論文0` → `統一記号表.md` → `出版計画_忘却論シリーズ.md`

## 8. よくある更新事故

以下は実際に発生した、または高い確率で発生する更新事故のパターンである。

### 8.1 記号表を飛ばして論文だけ直す

**事故内容**: 論文を改稿し、記号の意味や定理の内容を変更したが、`統一記号表.md` を更新しない。

**何が壊れるか**:
- 他の論文が旧定義の記号を参照し続ける → 論文間で同じ記号が異なる意味を持つ
- 定理安定度マップ (§2b) と実際の論文状態が乖離する
- 次の執筆者が §4b のチェックを行っても、記号表が stale なため誤った前提で書き始める

**復旧先**:
1. `drafts/infra/リファレンス/統一記号表.md` §1 で該当記号を更新
2. §1.12（衝突記号表）に新たな衝突が生じていないか確認
3. §2b（定理安定度マップ）の該当行を更新
4. §3（依存グラフ）に新たな依存が追加されていないか確認

### 8.2 レジストリ未更新で反論を書く

**事故内容**: 外部批判への反論を論文本文に直接書いたが、`批判反証レジストリ.md` にエントリを追加しない。

**何が壊れるか**:
- 同じ批判に別の執筆者が再度対応し、矛盾する反論が並立する
- 横断批判（Z-系）の場合、他の論文で同じ問題が再発する
- failure condition が不明なまま「解決済み」として扱われる

**復旧先**:
1. `drafts/infra/リファレンス/批判反証レジストリ.md` §2 に該当論文のエントリを追加
2. 横断批判の場合は §1 の Z-系エントリも確認
3. §1b のシリーズ共通ルールに照らして、応答が規則に従っているか確認
4. §4 の OP に新たな Open Problem が生じていないか確認

### 8.3 モノグラフ設計を公開系列の正本と誤認する

**事故内容**: `モノグラフ構成設計.md` の章立て・版番号を公開系列の正本として扱い、それに基づいて論文の版番号や順序を変更する。

**何が壊れるか**:
- モノグラフ設計は内部設計書であり、公開系列の配列を上書きしない（§5 取り違え参照）
- 版番号の正本は各論文ファイルのヘッダであり、モノグラフ設計の §4 版表は参照用
- 公開系列の番号と順序の正本は `plans/出版計画_忘却論シリーズ.md`

**復旧先**:
1. `plans/出版計画_忘却論シリーズ.md` で正本の番号と順序を再確認
2. 各論文ファイルのヘッダで版番号を直接確認
3. モノグラフ設計 §4 の版表を論文ヘッダに合わせて修正
